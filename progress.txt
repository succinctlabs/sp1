## Codebase Patterns
- MemoryGlobalChip events must be sorted by address before GPU transfer. The kernel reads events[i-1].addr for prev_addr, making it fully parallelizable (no sequential dependency).
- For chips with sequential dependencies (prev_addr from previous row), pass the initial value as a kernel parameter and have each thread look up the adjacent event from the sorted events array.
- IsZeroOperation: if a == 0 then inverse=0, result=1; else inverse=a.reciprocal(), result=0. Use populate_is_zero_from_field for field element input.
- LtOperationUnsigned (address comparison): iterate u16 limbs from MSB to LSB to find first differing pair. Set flag, compute field inverse of diff, store comparison limbs. bit = comparison result (always 1 for strictly increasing addresses).
- JTypeRecord is the simplest record type: only op_a (register write) + op_b/op_c (immediates). No register reads for b or c.
- For field inverse computation in CUDA, use `kb31_t::reciprocal()` method which computes `a^(p-2) mod p` for Koala Bear field
- SyscallInstrsChip uses RTypeReader (all 3 operands are registers). The syscall type is determined from prev_value of op_a (register t0). Need both prev_value (syscall code) and value (result) from memory record, so add a_value to the GPU event struct since GpuMemoryAccess only stores prev_value.
- For syscall test events, clk must satisfy clk ≡ 1 (mod 8) because CPUState.populate computes (clk_0_16 - 1) / 8 which underflows if clk_0_16 == 0. Use base_timestamp ≡ 1 (mod 8) and increment by 8.
- When cbindgen can't handle constant expressions like `[T; WORD_SIZE / 2]` or `[T; LONG_WORD_BYTE_SIZE]`, manually define the entire struct hierarchy in the CUDA file
- For complex operations with multiple array sizes (like MulOperation), may need to manually define MulCols, MulOperation, and U16toU8Operation together
- When cbindgen can't handle constant expressions like `[T; WORD_SIZE / 2]`, manually define the struct in the CUDA file
- ALUTypeRecord uses optional mem_c (Option<MemoryRecordEnum>) for immediate vs register mode
- For GPU event structs, flatten Option types to always include the field with a flag (is_imm) to indicate validity
- Test helper `test_traces_eq` compares CPU and GPU traces row by row and reports mismatches
- ITypeRecord is simpler than R/ALUTypeRecord: op_c is always an immediate (no mem_c needed)
- Chips using identical instruction formats can share GPU event structs via type aliases (e.g., SubGpuEvent = AddGpuEvent)
- SubwChip uses RTypeReader (like SubChip), not ALUTypeReader (like AddwChip) - check column struct definitions carefully
- **CRITICAL:** When chaining field arithmetic like `a * b - c + d`, use in-place operators (`*=`, `-=`, `+=`) to stay in `kb31_t` domain. The expression `a * b` returns `accel_t` (lazy 64-bit), and subsequent `accel_t` arithmetic can produce wrong results. Always force full Montgomery reduction between operations.
- BranchChip uses ITypeRecord where BOTH op_a and op_b are register reads (MemoryReadRecord), not writes. This differs from ALU chips where op_a is typically a write.
- JalChip uses JumpEvent (not JalEvent!) with JTypeRecord. The event tuple is `(JumpEvent, JTypeRecord)` stored in `jal_events`.
- JalChip is structurally very similar to UTypeChip: both use JTypeReader adapter with JTypeRecord. The key difference is JAL has two AddOperations (next_pc and return_addr) instead of UType's addend + add_operation + is_auipc.

  - **JTypeReader vs ITypeReader:** JTypeReader has `op_b_imm` and `op_c_imm` (both Word<T>) instead of ITypeReader's `op_b`/`op_b_memory` (register read) + `op_c_imm`. The UType adapter is simpler because both operands are program immediates.

  - **op_a_0 handling:** When `op_a == 0` (x0 register), addend is zero and add_operation.value is zero regardless of opcode. The is_auipc flag is still set correctly.
  - **Performance:** 62x GPU speedup on 1000 events test.

## 2026-01-29 - SyscallInstrsChip GPU Tracegen
- **What was implemented:** GPU trace generation for SyscallInstrsChip (ECALL instruction handling: HALT, ENTER_UNCONSTRAINED, HINT_LEN, COMMIT, COMMIT_DEFERRED_PROOFS, and other syscalls)
- **Files changed:**
  - `crates/sys/src/riscv_events.rs` - Added `SyscallInstrsGpuEvent` struct with a_value field for write record's current value
  - `crates/sys/build.rs` - Added cbindgen export for SyscallInstrsGpuEvent
  - `lib/tracegen/riscv/syscall_instrs.cu` - New CUDA kernel with manually-defined IsZeroOperation, SP1FieldWordRangeChecker, U16toU8Operation, and SyscallInstrColumns structs
  - `lib/tracegen/CMakeLists.txt` - Added syscall_instrs.cu to build
  - `crates/sys/src/tracegen.rs` - Declared extern fn for syscall_instrs kernel
  - `crates/cuda/src/tracegen.rs` - Added TracegenRiscvSyscallInstrsKernel trait
  - `crates/tracegen/src/riscv/syscall.rs` - Full CudaTracegenAir implementation with test and timing
- **Learnings:**
  - **Dual-value memory records:** SyscallInstrsChip needs BOTH prev_value (for syscall_id extraction) and value (for op_a_value column) from op_a's MemoryRecordEnum. Since GpuMemoryAccess only stores prev_value (from previous_record()), added `a_value` field to the GPU event for the Write record's current value.
  - **IsZeroOperation for syscall type detection:** Each syscall type is detected via IsZeroOperation on `(syscall_id - target_id)`. If the difference is zero, result=1, inverse=0. Otherwise result=0, inverse=a^(-1). Uses `kb31_t::reciprocal()` for field inverse.
  - **SP1FieldWordRangeChecker:** Contains a U16CompareOperation with just a `bit: T` field. Checks if ms_limb (limb[1]) < TOP_LIMB (32512 = 127*256). Used for HALT (op_b range check) and COMMIT_DEFERRED_PROOFS (op_c range check).
  - **U16toU8Operation:** Extracts low byte from each u16 limb. Used for `a_low_bytes` which decomposes the prev_value to extract the syscall_id byte.
  - **next_pc encoding for HALT:** When syscall is HALT, next_pc = [1, 0, 0] (HALT_PC = 1). Otherwise next_pc = [pc_low + 4, pc_mid, pc_high].
  - **COMMIT index_bitmap:** digest_idx comes from `record.b.value()` which for a read record equals prev_value. Only lower 3 bits used (index 0-7 into PV_DIGEST_NUM_WORDS array).
  - **COMMIT expected_public_values_digest:** 4 bytes from `record.c.value() as u32` in little-endian order.
  - **clk constraint:** Test events must have clk ≡ 1 (mod 8) for CPUState.populate. Using base_timestamp = 0x1_0000_1001 and increment of 8 satisfies this.
  - **Performance:** 65.47x GPU speedup on 1000 events test.

## 2026-01-29 - ByteChip & RangeChip GPU Tracegen
- **What was implemented:** GPU trace generation for ByteChip (byte lookup table) and RangeChip (range lookup table). These are lookup table chips with a fundamentally different pattern from instruction chips - they populate multiplicities from a HashMap rather than processing sequential events.
- **Files changed:**
  - `crates/sys/src/riscv_events.rs` - Added `ByteLookupGpuEntry` and `RangeLookupGpuEntry` structs
  - `crates/sys/build.rs` - Added cbindgen exports for both entry types
  - `lib/tracegen/riscv/lookup.cu` - New CUDA kernels for scatter-write multiplicity population
  - `lib/tracegen/CMakeLists.txt` - Added lookup.cu to build
  - `crates/sys/src/tracegen.rs` - Declared extern fns for both kernels
  - `crates/cuda/src/tracegen.rs` - Added TracegenRiscvByteLookupKernel and TracegenRiscvRangeLookupKernel traits
  - `crates/tracegen/src/riscv/lookup.rs` - Full CudaTracegenAir implementations and tests

- **Learnings:**
  - **Lookup table chips use scatter-write pattern:** Unlike instruction chips that process dense event arrays, lookup chips convert a HashMap into a flat array of entries and scatter-write multiplicities to specific (row, column) positions. The trace is mostly zeros.
  - **ByteChip structure:** 6 multiplicity columns (AND, OR, XOR, U8Range, LTU, MSB), 65536 rows (256x256 byte pairs). Row index = `(b << 8) + c`, column = opcode index.
  - **RangeChip structure:** 1 multiplicity column, 131072 rows (2^17). Row index = `a + (1 << b)` where `a < 2^b` and `b` ranges 0-16.
  - **byte_lookups uses hashbrown::HashMap:** The ExecutionRecord's `byte_lookups` field is `hashbrown::HashMap`, not `std::HashMap`. In tests, use `ByteRecord::add_byte_lookup_event()` to populate the record instead of constructing a HashMap directly.
  - **u16 shift overflow:** When generating test data for RangeChip, `1u16 << 16` overflows. Use `1u32 << bits` instead.
  - **Modest GPU speedup expected:** These chips are memory-bandwidth-limited scatter operations, not compute-intensive. ByteChip: 3.38x, RangeChip: 1.27x. The real benefit is keeping the data on GPU for subsequent proving operations.

## 2026-01-29 - MemoryGlobalChip GPU Tracegen
- **What was implemented:** GPU trace generation for MemoryGlobalChip (Init and Finalize variants). This chip proves that memory addresses are strictly increasing within each shard, using LtOperationUnsigned for address comparison and IsZeroOperation for zero-checking.
- **Files changed:**
  - `crates/sys/src/riscv_events.rs` - Added `MemoryGlobalGpuEvent` struct (addr, value, timestamp)
  - `crates/sys/build.rs` - Added cbindgen export for MemoryGlobalGpuEvent
  - `lib/tracegen/riscv/memory_global.cu` - New CUDA kernel with manually-defined MemoryInitCols, LtOperationUnsigned, IsZeroOperation, U16CompareOperation structs
  - `lib/tracegen/CMakeLists.txt` - Added memory_global.cu to build
  - `crates/sys/src/tracegen.rs` - Declared extern fn for memory_global kernel
  - `crates/cuda/src/tracegen.rs` - Added TracegenRiscvMemoryGlobalKernel trait
  - `crates/tracegen/src/riscv/memory_state.rs` - Full CudaTracegenAir implementation for MemoryGlobalChip with Init and Finalize tests

- **Learnings:**
  - **Sorting on CPU, parallel on GPU:** Events are sorted by address on CPU before GPU transfer. Each GPU thread reads events[i-1].addr for prev_addr, making the kernel fully parallelizable with no sequential dependency.
  - **previous_addr as kernel parameter:** The previous shard's last address is passed as a u64 kernel parameter (5th argument) rather than as part of the events array. Thread 0 uses this for its prev_addr.
  - **IsZeroOperation for field elements:** `is_prev_addr_zero` uses `populate_from_field_element` on the sum of the 3 prev_addr limbs. The GPU implementation uses `kb31_t::reciprocal()` for the field inverse when the sum is non-zero.
  - **LtOperationUnsigned for address comparison:** Always called with a_u64=1 (since addresses are strictly increasing). The comparison result bit is always 1. Iterates u16 limbs from MSB to LSB to find the first differing pair.
  - **prev_valid logic:** prev_valid = 0 only when prev_addr == 0 AND i != 0. This handles the edge case where the previous shard ended with address 0 (no memory initialized yet).
  - **Shared kernel for Init and Finalize:** Both variants use the same CUDA kernel - they differ only in which events and which public_values field is used (Rust side).
  - **Performance:** ~49x GPU speedup on 1000 events test (Init: 48.63x, Finalize: 49.19x).

## 2026-01-29 - ProgramChip GPU Tracegen
- **What was implemented:** GPU trace generation for ProgramChip, which has both preprocessed (16 columns) and main (1 column) traces. The preprocessed trace encodes the program's instructions (PC + opcode + operands), and the main trace stores instruction execution multiplicities.
- **Files changed:**
  - `crates/sys/src/riscv_events.rs` - Added `ProgramGpuInstruction` struct (opcode, op_a, op_b, op_c, op_a_0, imm_b, imm_c)
  - `crates/sys/build.rs` - Added cbindgen export for ProgramGpuInstruction
  - `lib/tracegen/riscv/program.cu` - New CUDA kernel with two kernels: preprocessed and main
  - `lib/tracegen/CMakeLists.txt` - Added program.cu to build
  - `crates/sys/src/tracegen.rs` - Declared extern fns for both kernels
  - `crates/cuda/src/tracegen.rs` - Added TracegenRiscvProgramPreprocessedKernel and TracegenRiscvProgramKernel traits
  - `crates/tracegen/src/riscv/program.rs` - Full CudaTracegenAir implementation with both preprocessed and main trace generation
  - `crates/tracegen/src/riscv/mod.rs` - Added preprocessed trace dispatch for ProgramChip in RiscvAir

- **Learnings:**
  - **Two traces:** ProgramChip is unique among RISC-V chips because it has both a preprocessed trace (static, program-dependent) and a main trace (dynamic, execution-dependent). Other chips only have main traces.
  - **Preprocessed trace (16 cols):** ProgramPreprocessedCols contains pc[3] + InstructionCols (opcode, op_a, op_b[4], op_c[4], op_a_0, imm_b, imm_c). InstructionCols has Word<T> fields for op_b and op_c (4 x u16 limbs each = 8 total), making 3 + 13 = 16 columns.
  - **Main trace (1 col):** Just multiplicity counts. Built on CPU by iterating all event vectors to count PC occurrences, then passed as flat u32 array to GPU for simple conversion to field elements.
  - **Padding row behavior:** For padding rows (i >= nb_instructions), the CPU uses index 0 (first instruction). The GPU kernel mirrors this: `uintptr_t idx = (i < nb_instructions) ? i : 0`.
  - **pc_base as kernel parameter:** The program counter base is passed as a 5th kernel argument (u64). Each instruction's PC = pc_base + idx * 4.
  - **Ambiguous Self::Program:** In RiscvAir dispatch, `Self::Program` is ambiguous (enum variant vs associated type). Use `<Self as MachineAir<F>>::Program` for the associated type.
  - **Performance:** Preprocessed: 27.19x speedup, Main: 35.04x speedup on 1000 instructions. Integration test: 55 shards, 6865 kHz.
