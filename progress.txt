## Codebase Patterns
- When cbindgen can't handle constant expressions like `[T; WORD_SIZE / 2]` or `[T; LONG_WORD_BYTE_SIZE]`, manually define the entire struct hierarchy in the CUDA file
- For complex operations with multiple array sizes (like MulOperation), may need to manually define MulCols, MulOperation, and U16toU8Operation together
- When cbindgen can't handle constant expressions like `[T; WORD_SIZE / 2]`, manually define the struct in the CUDA file
- ALUTypeRecord uses optional mem_c (Option<MemoryRecordEnum>) for immediate vs register mode
- For GPU event structs, flatten Option types to always include the field with a flag (is_imm) to indicate validity
- Test helper `test_traces_eq` compares CPU and GPU traces row by row and reports mismatches
- ITypeRecord is simpler than R/ALUTypeRecord: op_c is always an immediate (no mem_c needed)
- Chips using identical instruction formats can share GPU event structs via type aliases (e.g., SubGpuEvent = AddGpuEvent)
- SubwChip uses RTypeReader (like SubChip), not ALUTypeReader (like AddwChip) - check column struct definitions carefully

## 2026-01-27 - AddwChip GPU Tracegen
- **What was implemented:** GPU trace generation for AddwChip (32-bit add with sign extension)
- **Files changed:**
  - `crates/sys/src/riscv_events.rs` - Added `AddwGpuEvent` struct
  - `crates/sys/build.rs` - Added cbindgen exports for AddwCols, ALUTypeReader, AddwOperation, U16MSBOperation
  - `lib/tracegen/riscv/addw.cu` - New CUDA kernel for AddwChip
  - `lib/tracegen/CMakeLists.txt` - Added addw.cu to build
  - `crates/sys/src/tracegen.rs` - Declared extern fn for addw kernel
  - `crates/cuda/src/tracegen.rs` - Added TracegenRiscvAddwKernel trait
  - `crates/tracegen/src/riscv/alu.rs` - Implemented CudaTracegenAir for AddwChip, added tests with timing

- **Learnings:**
  - **cbindgen constant expressions:** cbindgen cannot resolve Rust constant expressions like `WORD_SIZE / 2`. The `AddwOperation` struct uses `[T; WORD_SIZE / 2]` which cbindgen only generates as a forward declaration. Solution: manually define the struct in the CUDA file.
  - **ALUTypeRecord vs RTypeRecord:** AddwChip uses ALUTypeRecord which differs from RTypeRecord in that `c` is an Option (None for immediate mode). The GPU event flattens this by always including `mem_c` field and using `is_imm` flag.
  - **U16MSBOperation:** AddwOperation includes a U16MSBOperation field to track the MSB of the 32-bit result for sign extension.
  - **ADDW semantics:** ADDW performs 32-bit addition on the lower 32 bits of operands and sign-extends the result to 64 bits.

- **Learnings:**

  - **ADDI semantics:** ADDI adds a 64-bit register value to a sign-extended 12-bit immediate. Result is written to the destination register.
  - **Test pattern reuse:** The same test structure (generate_*_events, compare traces) works well across different instruction types.

## 2026-01-27 - SubChip GPU Tracegen
- **What was implemented:** GPU trace generation for SubChip (64-bit subtraction)
- **Files changed:**
  - `crates/sys/src/riscv_events.rs` - Added `SubGpuEvent` as type alias to `AddGpuEvent`
  - `crates/sys/build.rs` - Added cbindgen exports for SubCols, SubOperation
  - `lib/tracegen/riscv/sub.cu` - New CUDA kernel for SubChip
  - `lib/tracegen/CMakeLists.txt` - Added sub.cu to build
  - `crates/sys/src/tracegen.rs` - Declared extern fn for sub kernel
  - `crates/cuda/src/tracegen.rs` - Added TracegenRiscvSubKernel trait
  - `crates/tracegen/src/riscv/alu.rs` - Implemented CudaTracegenAir for SubChip, added tests with timing

- **Learnings:**
  - **SubChip mirrors AddChip:** SubChip uses the same RTypeRecord as AddChip. The only difference is the operation: subtraction instead of addition.
  - **Type alias for GPU events:** Since SubChip and AddChip have identical event structures, we use a type alias (`SubGpuEvent = AddGpuEvent`) to avoid code duplication.
  - **SubOperation vs AddOperation:** SubOperation only contains `value: Word<T>` (result of b - c). It's structurally simpler than AddwOperation which has additional fields for sign extension.
  - **SUB semantics:** SUB performs 64-bit subtraction (b - c) with wrapping semantics. Result is written to the destination register.

## 2026-01-27 - MulChip GPU Tracegen

- **Files changed:**
  - `crates/sys/src/riscv_events.rs` - Added `MulGpuEvent` struct with opcode field to distinguish MUL variants
  - `crates/sys/build.rs` - Added cbindgen export for MulGpuEvent (MulCols and MulOperation manually defined)
  - `lib/tracegen/riscv/mul.cu` - New CUDA kernel for MulChip with manual struct definitions
  - `lib/tracegen/CMakeLists.txt` - Added mul.cu to build
  - `crates/sys/src/tracegen.rs` - Declared extern fn for mul kernel
  - `crates/cuda/src/tracegen.rs` - Added TracegenRiscvMulKernel trait
  - `crates/tracegen/src/riscv/alu.rs` - Implemented CudaTracegenAir for MulChip, added tests with timing

- **Learnings:**
  - **MulOperation complexity:** MulOperation has arrays sized by LONG_WORD_BYTE_SIZE (16) and WORD_SIZE (4), making cbindgen unable to resolve them. Had to manually define U16toU8Operation, MulOperation, and MulCols in the CUDA file.
  - **Opcode field in GPU event:** MulChip handles 5 different opcodes (MUL, MULH, MULHU, MULHSU, MULW). Added an opcode field to MulGpuEvent and an enum in CUDA to map them (MUL=0, MULH=1, etc.).
  - **Full 128-bit multiplication:** The multiplication kernel implements byte-by-byte multiplication with sign extension for signed variants, producing a 16-byte product array with carry propagation.
  - **Sign extension logic:** MULH sign-extends both operands, MULHSU sign-extends only the first operand (signed x unsigned), MULHU doesn't sign-extend either.
  - **MULW product MSB:** For MULW (32-bit multiply), need to compute the MSB of the 32-bit result (bit 31) for sign extension to 64-bits.

## 2026-01-28 - DivRemChip GPU Tracegen
- **What was implemented:** GPU trace generation for DivRemChip (division and remainder operations: DIV, DIVU, REM, REMU, DIVW, DIVUW, REMW, REMUW)
- **Files changed:**
  - `crates/sys/src/riscv_events.rs` - Added `DivRemGpuEvent` struct with opcode field to distinguish DivRem variants
  - `crates/sys/build.rs` - Added cbindgen export for DivRemGpuEvent (DivRemCols manually defined in CUDA)
  - `lib/tracegen/riscv/divrem.cu` - Comprehensive CUDA kernel with extensive manual struct definitions
  - `lib/tracegen/CMakeLists.txt` - Added divrem.cu to build
  - `crates/sys/src/tracegen.rs` - Declared extern fn for divrem kernel
  - `crates/cuda/src/tracegen.rs` - Added TracegenRiscvDivRemKernel trait
  - `crates/tracegen/src/riscv/alu.rs` - Implemented CudaTracegenAir for DivRemChip, added test with timing

- **Learnings:**
  - **Complex struct hierarchy:** DivRemCols requires many nested operation structs (IsZeroOperation, IsZeroWordOperation, IsEqualWordOperation, LtOperationUnsigned, MulOperation, AddOperation, U16MSBOperation). All must be manually defined in CUDA since cbindgen can't handle the constant expression array sizes.
  - **Division by zero handling:** When c=0, quotient = -1 (or UINT64_MAX for unsigned), remainder = b. Special handling needed for all 8 opcode variants.
  - **Overflow detection:** For signed division, check for MIN_VALUE / -1 overflow case (result would overflow). Implemented via IsEqualWordOperation comparisons.
  - **Signed vs unsigned computation:** Need separate logic for signed 64-bit (DIV/REM), unsigned 64-bit (DIVU/REMU), signed 32-bit word (DIVW/REMW), and unsigned 32-bit word (DIVUW/REMUW).
  - **Constraint verification:** c * quotient + remainder = b (in computational form). Need to compute carry propagation for the addition check across 8 u16 limbs.
  - **Remainder bound check:** |remainder| < |c| (when c != 0). Uses LtOperationUnsigned to verify this constraint.
  - **Network issues:** Build and test blocked by proxy restrictions on crates.io access. Code is syntactically verified via rustfmt --edition 2021.
